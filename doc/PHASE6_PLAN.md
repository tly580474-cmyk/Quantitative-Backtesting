# 量化回测第六阶段：因子研究与多因子评价平台

## 0. 当前决策

第六阶段的最终目标保持为“完成可复现的因子研究与多因子评价平台”，但实施顺序先调整为
数据底座优先：

1. 先解决 DuckDB/Parquet 研究快照与 MySQL 权威库的数据同步问题，也就是快照追平；
2. 先解决 MySQL 与当前研究快照的一致备份、校验和恢复演练问题；
3. 运维界面暂时忽略，不作为第六阶段前半段交付物；
4. 在上述底座稳定后，再继续因子研究、报告和多因子合成能力。

截至 2026-07-07，5.5 阶段已经完成一次真实追平与备份校验：

- 当前研究快照：`b8a23bce-2f65-46c5-9614-4edc6f359df5-20260707040738`
- 数据追平至：`2026-07-07`
- 快照行数：17,047,012
- 最新备份：`server/data/backups/backup-20260707041121`
- `snapshot:verify`、`snapshot:benchmark`、`backup:verify`、`backup:restore-check` 均已通过。

因此第六阶段不再把“运维界面”作为近期重点，而是把“自动追平、可验证备份、恢复演练、
研究任务引用固定快照”作为 P0 验收线。

## 1. 阶段定位

第六阶段承接 5.5 阶段的全量历史行情库、Parquet 快照和 DuckDB 研究查询能力，
建设面向 A 股日频截面的因子研究平台。系统应支持用户定义、计算、评价和比较
技术类与行情衍生类因子，并通过 IC、Rank IC、分层收益、换手和样本外验证判断
因子是否具有稳定解释力。

核心闭环：

```text
选择数据版本 / 股票池 / 研究区间
        ↓
定义因子与预处理规则
        ↓
DuckDB 批量计算截面因子值
        ↓
去极值 / 标准化 / 中性化 / 可交易性过滤
        ↓
IC / 分层收益 / 衰减 / 换手 / 成本分析
        ↓
保存因子版本与研究报告
        ↓
进入多因子合成或策略回测复核
```

本阶段优先解决“一个因子是否稳定有效，以及多个因子如何可复现地比较”。不进入
实盘交易、分钟级高频因子和自动机器学习挖掘。

## 2. 进入条件

只有满足 5.5 阶段门槛后才开始本阶段正式开发：

1. 全量历史数据已发布且通过基线核对；
2. 至少连续 5 个交易日增量更新无重复、无漏日；
3. MySQL 与 Parquet 快照能够自动追平；
4. 全市场研究查询不会影响普通页面；
5. 回测结果能够固定引用数据版本；
6. 已完成数据库备份与恢复演练。

若任一条件未满足，第六阶段只允许进行接口草图和 Mock 原型，不接入真实全市场计算。

## 3. 阶段目标

1. 建立因子定义、版本、依赖字段和计算结果的统一模型；
2. 支持行情衍生因子和常用技术因子的批量截面计算；
3. 支持股票池、行业、上市天数、ST、停牌、涨跌停和流动性过滤；
4. 支持去极值、缺失值处理、标准化及行业/市值中性化；
5. 支持 IC、Rank IC、分层收益、因子衰减、换手率和成本敏感性评价；
6. 支持单因子报告、多因子相关性分析和加权合成；
7. 研究结果必须固定引用数据快照、因子版本、股票池版本和评价配置；
8. 提供可复现导出，方便后续接入策略实验室和组合回测。

## 4. 范围与优先级

### 4.1 P0：因子协议与研究数据集

- 定义 `FactorDefinition`、`FactorVersion`、`FactorRun` 和 `FactorReport`；
- 因子表达式必须是白名单 DSL，不执行任意 JavaScript 或 SQL 片段；
- 因子依赖字段、预热期、滞后期和可用日期范围必须显式声明；
- 研究任务固定引用 Parquet 快照 manifest，不直接扫未发布数据；
- 股票池支持全 A、沪深主板、创业板、科创板、北交所和自定义列表；
- 建立基础行业分类、上市天数、ST、成交额、停牌和涨跌停过滤口径；
- 所有收益标签默认使用未来收益，因子值与未来收益之间必须严格错期。

### 4.2 P1：单因子计算 MVP

- 提供因子列表、创建、复制、禁用和版本发布；
- 内置首批行情因子：动量、反转、波动率、换手率、成交额、量价背离、均线偏离；
- 内置首批技术因子：RSI、MACD、BOLL 位置、ATR、OBV、KDJ、CCI；
- 支持按交易日批量计算截面因子值；
- 支持按研究区间生成 1、5、10、20 日未来收益标签；
- 计算结果写入版本化 Parquet 产物，不把全量因子矩阵塞入 MySQL；
- MySQL 只保存任务、版本、摘要指标、报告索引和审计信息；
- 支持任务进度、取消、失败重试和运行日志。

### 4.3 P1：因子预处理与评价

- 支持 MAD 或分位数去极值；
- 支持 z-score、rank 标准化和缺失值策略；
- 支持按行业中性化和按对数市值中性化；
- 计算 Pearson IC、Spearman Rank IC、ICIR、胜率和 t 值；
- 计算分层收益、多空组合收益、最大回撤和换手率；
- 分析不同持有期下的因子衰减；
- 评价结果必须同时展示原始因子、预处理后因子和中性化后因子的差异；
- 交易成本分析至少支持单边费率和换手扣减。

### 4.4 P2：多因子研究与报告

- 支持因子相关性矩阵和聚类视图；
- 支持等权、IC 加权、RankIC 加权和手动权重合成；
- 支持训练区间确定权重，验证区间只做复核；
- 支持合成因子的分层收益、IC 和换手分析；
- 支持将合成因子导出为“选股信号候选”，供策略实验室或回测页面复核；
- 保存 HTML/Markdown 风格研究报告，包括配置、样本、图表和风险提示；
- 支持报告导出 JSON/CSV，避免大矩阵进入普通 Excel。

### 4.5 P2：股票池与时点化增强

- 建立股票池版本模型，记录成员生效日期和移除日期；
- 对行业分类、ST 状态、市值和可交易状态进行时点化引用；
- 预留指数历史成分接入位置，但本阶段 MVP 不强依赖外部指数成分数据；
- 预留财务因子接口，但不在缺少财报时点化和公告日期前计算财务因子；
- 提供幸存者偏差检查，报告样本覆盖率和被过滤证券数量。

## 5. 明确不纳入第六阶段

- 分钟线、Tick、盘口和实时因子；
- 实盘交易、券商接口和自动调仓；
- 自动爬取未授权财务、新闻、公告和另类数据；
- 遗传算法、贝叶斯优化、AutoML 和自动挖掘海量公式；
- 云端分布式计算、Spark、ClickHouse 或 Kafka；
- 未时点化的财务因子回测结论；
- 将研究结果包装为投资建议或保证收益的选股推荐。

## 6. 关键业务口径

### 6.1 时间与防泄漏

- 第 `T` 日因子只能使用 `T` 日收盘及以前已知数据；
- 未来收益标签从 `T+1` 开始计算，不允许使用当日收盘后不可成交价格制造收益；
- 调仓假设、持有期和收益计算口径必须写入报告；
- 中性化、市值分组和行业分组只能使用当日已知截面；
- 因子预热期可以读取研究起点前历史，但不得把预热期纳入评价样本。

### 6.2 股票池与可交易性

- 默认排除上市不足 60 个交易日、ST、停牌和一字涨跌停样本；
- 默认流动性过滤使用近 20 日日均成交额下限；
- 北交所样本是否纳入由股票池配置明确控制；
- 退市股票在其可交易历史内仍可进入样本，不得因当前退市而静默剔除；
- 每个交易日的有效样本数低于阈值时，该日评价标记为样本不足。

### 6.3 因子方向与缺失值

- 因子必须声明方向：值越大越好、值越小越好或待检验；
- 缺失值不得默认填 0；必须选择剔除、行业中位数填充或全市场中位数填充；
- 极值处理、中性化和标准化顺序必须固定并记录；
- 因子覆盖率、缺失率和极值截断比例进入报告首页；
- 所有评分和排序均可追溯到原始因子值和预处理配置。

## 7. 技术方案

### 7.1 模块划分

```text
server/src/factorResearch/
  definitions/
    schema.ts                 因子定义、表达式和版本协议
    builtins.ts               内置技术和行情因子
    validator.ts              DSL 校验与依赖字段检查
  engine/
    factorCompiler.ts         DSL 到 DuckDB 查询计划
    factorRunner.ts           因子批量计算任务
    preprocessing.ts          去极值、标准化和中性化
    labels.ts                 未来收益标签生成
    evaluator.ts              IC、分层收益、衰减和换手
    composite.ts              多因子合成
  repositories/
    factorRepository.ts       MySQL 元数据
    artifactRepository.ts     Parquet 报告与矩阵产物
  routes/
    factors.ts
    factorRuns.ts
    factorReports.ts

src/features/factorResearch/
  FactorResearchPage.tsx
  FactorDefinitionEditor.tsx
  FactorRunPanel.tsx
  FactorReportPage.tsx
  FactorLayerChart.tsx
  FactorIcChart.tsx
  FactorCorrelationMatrix.tsx
```

### 7.2 数据与产物

MySQL 保存轻量元数据：

```text
factor_definitions
  id, name, description, status, created_at, updated_at

factor_versions
  id, factor_id, version, expression, direction,
  dependencies, warmup_days, checksum, published_at

factor_runs
  id, factor_version_id, snapshot_id, universe_id, status,
  date_start, date_end, preprocessing_config, label_config,
  total_dates, completed_dates, artifact_uri, created_at, finished_at

factor_reports
  id, run_id, summary_metrics, report_uri, created_at
```

大矩阵保存到 Parquet：

```text
factor_values/
  snapshot=<snapshot_id>/factor=<factor_version_id>/year=YYYY/*.parquet

factor_labels/
  snapshot=<snapshot_id>/horizon=<N>/year=YYYY/*.parquet

factor_reports/
  run=<run_id>/summary.json
  run=<run_id>/layers.parquet
  run=<run_id>/ic_series.parquet
```

### 7.3 执行架构

```text
用户提交研究任务
    ↓
校验因子版本、股票池、快照和日期范围
    ↓
生成 DuckDB 查询计划
    ↓
按年份或日期分片计算因子值
    ↓
生成未来收益标签并错期合并
    ↓
执行预处理和评价
    ↓
写入 Parquet 产物与 MySQL 摘要
    ↓
前端读取摘要和分页图表数据
```

实施约束：

- 普通 API 不返回全量因子矩阵；
- 研究任务设置并发、超时和内存上限；
- 同一快照和同一因子版本的产物可复用，但配置变化必须生成新 run；
- DuckDB 临时文件目录、产物目录和清理策略可配置；
- 因子表达式编译结果需要记录 checksum，防止同名公式漂移；
- 因子报告页面使用服务端分页和聚合，不在浏览器计算全市场多年矩阵。

## 8. 页面与交互

桌面端采用四区工作流：

1. 因子库：查看内置和自定义因子，发布版本；
2. 研究配置：选择数据快照、股票池、区间、预处理和收益标签；
3. 运行监控：任务进度、分片日志、失败原因、取消和重试；
4. 研究报告：IC、分层收益、衰减、换手、相关性和样本覆盖。

交互要求：

- 因子编辑器必须实时展示依赖字段、预热期和可用样本预估；
- 启动全市场多年任务前展示预计交易日数、证券数和产物大小；
- 报告首页先展示结论摘要和风险标记，再展开详细图表；
- 图表必须有表格数据替代，颜色不能作为唯一状态表达；
- 多因子权重调整使用可解释控件，并展示权重、方向和相关性冲突；
- 移动端保留报告查看和任务状态，不强塞矩阵编辑体验；
- 页面必须明确展示“历史统计不构成投资建议”。

视觉延续现有金融分析工具风格：高信息密度、浅色、克制图表、清晰状态，不做营销式首页。

## 9. 实施顺序与工作量

### M0：数据底座冻结：快照追平与备份恢复（2～4 人日）

- 固化 MySQL 增量完成后重建受影响年份分区的快照追平流程；
- 每次追平后强制执行 `snapshot:verify`，必要时执行 `snapshot:benchmark`；
- 固化 `backup:create`、`backup:verify` 和 `backup:restore-check`，备份必须同时包含 MySQL dump 与当前研究快照；
- 补一轮恢复演练：临时库导入 dump、临时快照目录校验、行数和最大交易日对齐；
- 因子研究任务启动前必须读取并记录当前 snapshot id，不允许扫未发布数据；
- 记录失败处理：快照发布失败不得切换 current，备份校验失败不得进入研究任务。

### M1：口径冻结与样本基准（2～3 人日）

- 冻结因子协议、收益标签、股票池过滤和防泄漏规则；
- 建立 100 只证券、1 年数据的固定研究样本；
- 记录 DuckDB 截面计算、join 和分层聚合基线；
- 输出首批内置因子清单和人工核对样例。

### M2：因子定义与计算 MVP（4～6 人日）

- 新增因子元数据表、版本表和 API；
- 实现白名单因子 DSL、校验器和内置因子；
- 实现 DuckDB 因子计算任务和 Parquet 产物写入；
- 完成样本数据端到端计算。

### M3：预处理与单因子评价（5～7 人日）

- 实现未来收益标签、错期合并和覆盖率统计；
- 实现去极值、标准化、行业/市值中性化；
- 实现 IC、Rank IC、ICIR、分层收益、换手和成本扣减；
- 生成单因子报告摘要和图表数据。

### M4：前端研究工作台（3～5 人日）

- 因子库、因子编辑器和版本发布；
- 研究任务配置、运行监控和失败重试；
- 单因子报告页面、IC 曲线、分层收益和衰减图；
- 报告导出和任务历史。

运维界面暂缓；只保留面向研究人员的因子工作台和报告查看能力。

### M5：多因子合成与样本外验证（5～7 人日）

- 因子相关性矩阵；
- 等权、IC 加权、RankIC 加权和手动合成；
- 训练/验证区间切分和样本外复核；
- 合成因子报告与回测候选导出。

### M6：性能、治理与正式验收（3～5 人日）

- 全市场 3 年、5 年、10 年研究基准；
- 并发限制、任务取消、临时文件清理和产物复用；
- 样本偏差、覆盖率和数据新鲜度风险提示；
- 文档、运维手册、备份恢复和回归测试。

预计总工作量：25～38 人日。建议先交付 M0～M4，形成“数据可追平、可恢复、单因子可研究”
的闭环，再进入多因子合成。

## 10. 测试计划

### 10.1 单元测试

- 因子 DSL 解析、依赖字段和非法表达式拒绝；
- 内置因子与前端指标计算口径抽样一致；
- 未来收益标签错期、持有期和边界日期；
- 去极值、标准化、中性化和缺失值策略；
- IC、Rank IC、ICIR、分层收益和换手计算；
- 因子方向、覆盖率和样本不足标记；
- 多因子权重合成和相关性计算。

### 10.2 集成测试

- Parquet 快照 → 因子计算 → 标签合并 → 报告产物；
- 同一配置重复运行产物 checksum 一致；
- 因子版本变化后生成新 run，不覆盖旧报告；
- 任务取消后可重试且不读取半成品；
- MySQL 摘要与 Parquet 产物行数、日期范围一致；
- 前端报告分页读取不触发全量矩阵下载；
- 数据快照缺失或滞后时阻止研究任务启动。

### 10.3 性能与稳定性

- 100 只、1,000 只、全市场三档研究基准；
- 1 年、3 年、10 年三档日期跨度基准；
- 10 个内置因子批量计算的峰值内存和临时文件大小；
- 两个研究任务并发时普通行情页面仍可用；
- 大报告打开、分页、排序和图表切换无明显卡顿；
- 磁盘空间不足、DuckDB 查询超时和产物写入失败的恢复。

## 11. 验收标准

- [ ] 可以创建、校验、发布和禁用因子版本；
- [ ] 内置行情因子和技术因子可在固定快照上批量计算；
- [ ] 因子值、未来收益和评价样本严格错期，无未来函数；
- [ ] 股票池、ST、停牌、涨跌停、上市天数和流动性过滤可配置且可追溯；
- [ ] 去极值、标准化、中性化和缺失值处理进入报告配置；
- [ ] IC、Rank IC、分层收益、衰减、换手和成本分析可用；
- [ ] 单因子报告展示覆盖率、样本不足和偏差风险；
- [ ] 多因子相关性和至少两种合成方式可用；
- [ ] 研究结果固定引用数据快照、因子版本、股票池版本和评价配置；
- [ ] MySQL 权威库与 DuckDB/Parquet 研究快照可重复追平，并能阻断滞后快照进入研究任务；
