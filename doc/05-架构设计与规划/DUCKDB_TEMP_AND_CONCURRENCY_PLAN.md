# DuckDB 临时目录与并发治理计划

## 背景与结论

服务端多个生产入口使用 `:memory:` DuckDB，但没有显式配置 `temp_directory`。全市场窗口、
排序、聚合和联表超过内存上限后会写入进程工作目录下共享的 `.tmp`。锁定测试还会并行
启动相关性和衰减审计，因此不同实例可能同时创建、使用或删除同一目录。一次审计失败时，
`Promise.all` 中的另一个审计仍会继续运行，形成不可见的 CPU 与临时磁盘占用。

## 目标

1. 每个 DuckDB 实例拥有独立、可追踪、可安全清理的临时目录。
2. 服务进程内所有重型 DuckDB 工作共享并发上限，避免 CPU、内存和磁盘过载。
3. 基础设施故障不再被误判为候选因子质量失败。
4. 后续减少重复扫描和窗口计算，缩短锁定测试耗时。

## 分阶段计划

### P0：临时目录隔离（本轮）

- 新增统一 DuckDB 运行时，使用 `mkdtemp` 为每个实例创建唯一目录。
- 通过 `temp_directory` 和 `max_temp_directory_size` 显式传给 DuckDB。
- 临时根目录默认位于操作系统临时目录，可由 `DUCKDB_TEMP_ROOT` 覆盖。
- 实例关闭后只清理自身目录，不允许删除共享目录。
- 接入因子回测、组合回测、研究查询、快照构建/校验和挖掘调度。

验收：两个实例并存时目录不同；关闭一个实例不会影响另一个继续查询。

### P0：全局并发控制（本轮）

- 统一运行时提供 FIFO 信号量，默认最多 2 个 DuckDB 实例并行。
- 可通过 `DUCKDB_MAX_CONCURRENT` 调整，合法范围 1～8。
- 锁定测试的相关性与衰减审计改为串行，避免同一候选重复争抢资源。
- 暴露只读运行时统计，便于后续健康检查和监控。

验收：超过上限的实例进入队列；实例异常关闭也必须释放名额。

### P1：可取消和可重试

- 为后台锁定测试建立运行上下文，失败或超时时调用 `connection.interrupt()`。
- 引入 `test_failed`，或将基础设施失败恢复为 `frozen`；保留错误详情和重试入口。
- 服务启动时清理没有活动所有者的历史临时目录。

### P1：查询复用与性能优化

- 单次锁定测试复用一个 DuckDB 会话。
- 将 `scored` / `analysis_rows` 物化一次，汇总、分层、相关性和衰减复用结果。
- 评估线程数与内存配比，记录峰值内存、临时磁盘量和各阶段耗时。

### P2：运维与容量保护

- 增加临时磁盘软硬水位、队列长度和运行时长告警。
- API 展示 `queued/materializing/backtesting/auditing` 阶段。
- 增加异常退出、磁盘不足、强制溢写和并发候选的集成测试。

## 配置

| 配置 | 默认值 | 说明 |
| --- | --- | --- |
| `DUCKDB_TEMP_ROOT` | 系统临时目录下 `quant-backtest-duckdb` | DuckDB 独立临时目录的父目录 |
| `DUCKDB_MAX_CONCURRENT` | `2` | 服务进程内最大并行实例数 |
| `DUCKDB_MAX_TEMP_SIZE` | `50GB` | 单实例临时目录容量上限 |

## 回滚

统一运行时只封装实例创建、关闭和排队，不改变 SQL 结果。出现兼容问题时，可以逐入口恢复为
直接创建实例；因子值、快照和回测报告均不需要迁移。
