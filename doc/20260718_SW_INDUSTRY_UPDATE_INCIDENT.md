# 申万行业日线未更新调查记录（2026-07-18）

## 结论

`sw_industry_daily_bars` 停留在 2026-07-16，并非本地定时任务未运行，也不是数据库写入失败。2026-07-17 18:01 和 18:30 的任务均成功执行，但申万趋势接口当时返回的最新交易日仍为 2026-07-16；截至 2026-07-18 14:17 再次实测，官方源仍未发布 2026-07-17 行情。

调查同时发现申万当前元数据接口存在空响应及名称乱码。旧流程把元数据更新放在行情更新之前，因此元数据异常会阻断仍然可用的趋势行情接口，放大单一上游故障的影响。

## 证据

- `server/.logs/research-snapshot/research-update.log`：2026-07-17 两次 `sw-industry:update` 均成功，返回 31 个一级行业、140727 行行情，最大日期为 2026-07-16。
- 2026-07-18 手工执行 `npm run sw-industry:update`：趋势行情仍为 31 个行业、140727 行，最大日期仍为 2026-07-16。
- 数据库当前行业归属覆盖 5486/5486 个有效股票，一级行业数量 31，说明行业定义与成分数据未丢失。
- 权威日线及分钟湖均已到 2026-07-17，延迟仅发生在申万官方行业趋势源。

## 已实施修复

1. `sw_industry_update.py` 增加 `--bars-only`，允许直接使用数据库中已验证的行业定义更新行情。
2. 完整更新遇到元数据异常时，自动回退到数据库缓存定义，继续抓取并发布行业行情，不再整批失败。
3. 更新结果增加 `metadataFresh` 和 `warning`，明确区分“元数据异常”和“行情发布滞后”。
4. 数据健康门禁对覆盖完整且仅落后 1—3 个自然日的申万行情标记为 `warn`，超过窗口仍为 `fail`。
5. Windows 任务增加 08:30 早盘补偿，保留 18:00、18:30 两次盘后执行；2026-07-18 已重新登记。

## 当前状态与后续动作

当前数据健康门禁整体为 `warn`，唯一告警为申万行情落后 1 个自然日。系统会在 08:30、18:00、18:30 自动重试；上游发布 2026-07-17 数据后即可自动追平，无需手工改库。

若连续超过 3 个自然日仍未更新，门禁会恢复为失败，应进一步启用第二行业行情源或人工校验后补数；不得用个股收益临时合成后冒充申万官方行情。
