# 量化回测 5.5 阶段：全量历史行情库与高性能研究底座

## 1. 阶段定位

5.5 阶段承接第五阶段已经建立的证券、日线、复权、同步任务和质量治理能力，
将现有本地 A 股历史文件一次性迁入统一数据底座，并解决数千万行数据的导入、
查询、复权和研究计算压力。

本阶段不提前建设因子研究功能。核心目标是先让数据具备以下特征：

- MySQL 中存在可持续增量更新的权威事实层；
- 普通页面和单证券回测能够稳定、低延迟地读取数据；
- 全市场扫描不挤占 MySQL，由 Parquet + DuckDB 承担分析型负载；
- 导入、更新、复权和研究结果均可追溯、可校验、可重复执行。

```text
历史 CSV
   ↓ 流式解析、校验、标准化
MySQL 暂存 / 批量装载
   ↓ 原子发布
MySQL 权威事实层
   ├─ 单证券、单日截面、数据管理、增量更新
   └─ 生成版本化 Parquet 快照
                    ↓
               DuckDB 研究查询
                    ↓
          全市场回测 / 因子计算（后续阶段）
```

## 2. 已知数据基线

| --- | ---: | ---: | ---: |

| 前复权 | 5,825 | 17,681,063 | 4.80 GiB |
| 后复权 | 5,516 | 16,796,356 | 4.74 GiB |
| 合计 | 17,166 | 51,513,483 | 14.27 GiB |

质量基线：

- 5,477 只证券更新至 `2026-07-03`；
- 321 只证券已退市，26 只证券处于永久忽略或临时冷却状态；
- `000508` 三种口径均只有表头；
- 后复权缺少 309 个文件，主要为北交所证券；
- 前复权包含约 64.5 万条不复权目录不存在的早期交易记录；
- 部分早期估值字段为空，导入时必须保留 `NULL`，不得静默伪造。

以上数量作为首次全量导入的验收基线；后续源文件发生更新时，必须生成新的
导入批次和基线报告。

## 3. 阶段目标

### 3.1 P0：权威行情事实层

1. 将不复权行情作为生产环境的价格事实层；
2. 将证券静态信息、日线、每日估值和复权因子分表存储；
3. 避免将三套价格完整复制三遍；
4. 保留无法由不复权数据重建的早期前复权记录；
5. 所有大表使用紧凑数据类型、复合主键和明确的空值语义；
6. 全量导入幂等、可中断续传，并能从文件级或分片级恢复。

### 3.2 P0：批量导入与校验

1. 实现独立于 Web 服务启动过程的 CLI 导入器；
2. 流式读取 CSV，不将单目录或单证券全集一次性载入内存；
3. 使用 `LOAD DATA LOCAL INFILE` 或等价批量装载能力；
4. 记录导入批次、文件、校验和、行数、状态、耗时和错误；
5. 导入后核对总行数、证券行数、日期范围、重复键和抽样哈希；
6. 导入失败不发布半完成数据版本。

### 3.3 P1：高性能业务查询

1. 数据管理页使用服务端分页、筛选和排序；
2. 单证券日期区间查询使用聚簇主键范围扫描；
3. 单交易日全市场查询使用日期辅助索引；
4. 前复权和后复权由服务端按需生成，不在浏览器拼接；
5. API 设置日期跨度、返回行数和证券数量上限；
6. 大结果使用流式响应、任务结果或文件产物，不进入普通 JSON 接口。

### 3.4 P1：分析型查询隔离

1. 从已发布的 MySQL 数据版本生成 Parquet 快照；
2. Parquet 按年份分区，单文件目标大小控制在 128～512 MiB；
3. DuckDB 仅消费已发布快照，不直接修改权威数据；
4. 全市场扫描、横截面排名和组合回测优先使用 DuckDB；
5. MySQL 故障和 Parquet 快照滞后必须分别展示，不混淆状态。

## 4. 明确不纳入本阶段

- 分钟线、Tick、逐笔成交和盘口；
- 实盘交易、券商接口和账户管理；
- 财务报表、公告、新闻和舆情数据；
- IC、Rank IC、因子分层、中性化和多因子合成；
- 分布式计算集群、ClickHouse、Spark 或 Kafka；
- 将历史缺失估值字段自动填成未经验证的估算值；
- 在浏览器中加载或计算全市场多年数据。

## 5. 存储架构

### 5.1 MySQL 与 DuckDB 的职责

| 场景 | 执行引擎 |
| --- | --- |
| 证券搜索、状态和数据范围 | MySQL |
| 单证券多年 K 线 | MySQL |
| 单交易日全市场截面 | MySQL |
| 每日增量写入与修订 | MySQL |
| 数据版本、任务和质量问题 | MySQL |
| 全市场多年扫描 | DuckDB + Parquet |
| 横截面排序、滚动因子和组合回测 | DuckDB + Parquet |

MySQL 是唯一权威事实源；Parquet 是可重建的分析快照，不承接在线增量写入。

### 5.2 标识和数据类型

- 保留现有 `instruments.id` UUID 作为 API 公共标识；
- 为大表增加 `instrument_key INT UNSIGNED` 紧凑内部键；
- 交易日期使用 MySQL `DATE`，不使用 `VARCHAR(10)`；
- 成交量使用整数类型，金额和价格使用统一精度；
- `NULL` 表示来源缺失，数值 `0` 只表示真实的零；
- 来源、口径和状态使用短整型字典或短枚举，不在每行重复长字符串。

### 5.3 建议事实表

```text
instruments
  id, instrument_key, market, symbol, name, type,
  industry, list_date, delist_date, status

daily_bars_v2
  instrument_key, trade_date,
  open, high, low, close, previous_close,
  volume, amount, turnover_rate_pct,
  source_key, source_version, fetched_at
  PK (instrument_key, trade_date)
  INDEX (trade_date, instrument_key)

daily_stock_metrics
  instrument_key, trade_date,
  total_shares, float_shares,
  total_market_cap, float_market_cap,
  pe_ttm, pb, ps_ttm, volume_ratio,
  is_st, is_limit_up
  PK (instrument_key, trade_date)

adjustment_factors_v2
  instrument_key, effective_date, factor, price_offset,
  source_key, factor_version
  PK (instrument_key, effective_date, factor_version)

adjusted_bar_overrides
  instrument_key, trade_date, adjustment_mode,
  open, high, low, close, reason, source_batch_id
  PK (instrument_key, trade_date, adjustment_mode)

data_import_batches
  id, source_root, source_snapshot, status,
  total_files, completed_files, total_rows, imported_rows,
  started_at, finished_at, published_at

data_import_files
  batch_id, relative_path, adjustment_mode,
  checksum, expected_rows, imported_rows,
  min_date, max_date, status, error_message
```

首次迁移使用 `v2` 影子表，不直接破坏当前 `daily_candles`。完成校验和性能验收后，
Repository 再切换到新表；旧表至少保留一个发布周期。

## 6. 字段归属

### 6.1 直接保存

- 日线：日期、开高低收、前收盘、成交量、成交额、换手率；
- 每日指标：总股本、流通股本、总市值、流通市值、PE、PB、PS、量比；
- 状态：ST、涨停；
- 证券：代码、名称、市场、行业、上市日期、退市日期。

### 6.2 按需计算

- 涨幅、振幅；
- 3、6、10、25 日涨幅；
- 5、10、20、30、60、120、250 日均线；
- 前复权和后复权价格。

派生字段不进入权威事实表，避免历史修订后批量重写数千万行。确有性能需要时，
只能建立带版本号的缓存或 Parquet 派生列。

## 7. 复权迁移规则

1. 不复权 OHLCV 作为权威原始价格；
2. 在日期重叠区间，根据不复权和前复权价格推导或校验复权因子；
3. 复权参数按公司行为变化点压缩保存；现金分红使用
   `adjusted_price = raw_price × factor + price_offset`，不按交易日无意义重复；
4. 使用后复权文件交叉验证因子，不将其作为第二份权威事实；
5. 前复权独有的早期记录写入 `adjusted_bar_overrides`；
6. 无法验证的复权区间标记为 `warning`，不得宣称已精确重建；
7. 北交所缺少后复权文件不阻断不复权行情发布。

复权抽样验收要求：

- OHLC 重建后的绝对误差不超过该证券最小价格单位；
- 日期、成交量和成交额不得因复权发生变化；
- 除权日前后人工样例与源文件一致；
- 因子版本进入回测数据快照。

## 8. 全量导入流程

### 8.1 预检

1. 扫描三个目录并生成只读清单；
2. 记录文件大小、修改时间、SHA-256、行数和日期范围；
3. 检查表头、代码、日期、重复键、OHLC 关系和关键字段类型；
4. 将空文件、退市证券、忽略证券和缺失复权文件单独归类；
5. 预检通过后冻结本批次 `source_snapshot`。

### 8.2 装载

1. 先 upsert 证券主数据并分配 `instrument_key`；
2. 按文件流式转换为规范化 TSV 分片；
3. 单分片建议 25～50 万行或不超过 256 MiB；
4. 使用 MySQL 批量装载写入影子表；
5. 每完成一个分片立即记录 checkpoint；
6. 导入器退出后可从最后成功 checkpoint 恢复；
7. 初始装载完成后再建立非必要辅助索引。

禁止使用现有 500 行 Drizzle upsert 执行首次 1,700 万行装载。该路径继续用于每日
约 5,800 行的增量更新。

### 8.3 发布

1. 核对批次总数与源文件基线；
2. 执行重复键、空主键、日期越界和孤立证券检查；
3. 按证券抽样比较源文件与数据库；
4. 生成数据版本和质量报告；
5. 原子切换“当前已发布版本”；
6. 异步生成 Parquet 快照；
7. 快照校验完成后才允许 DuckDB 研究任务使用。

## 9. 查询与 API 约束

- 单证券 K 线默认最多返回 5,000 行；
- 列表分页使用稳定排序，深分页改用 keyset pagination；
- 禁止普通 API 返回“全部证券 × 全部日期”；
- 单日全市场查询只返回调用方声明的字段；
- 复权计算在服务端执行，并返回 `adjustmentMode` 和 `factorVersion`；
- 回测任务记录 MySQL 数据版本或 Parquet 快照 ID；
- 长查询必须进入后台任务队列，支持取消、超时和并发限制；
- MySQL 连接池继续服务在线请求，导入器使用独立连接和并发上限。

## 10. 开发里程碑

## 11. 性能预算

性能指标以项目实际部署机器、冷缓存和热缓存分别记录，不以开发机单次结果替代。

| 场景 | 目标 |
| --- | --- |
| 单证券 20 年日线 | P95 ≤ 200 ms |
| 单交易日全市场基础字段 | P95 ≤ 300 ms |
| 数据管理列表 50 条 | P95 ≤ 500 ms |
| 每日约 5,800 行增量 upsert | ≤ 60 s |
| 全市场一年基础列 DuckDB 扫描 | ≤ 5 s |
| 普通 API 最大响应 | ≤ 10 MiB |
| 导入期间健康检查 | 持续可用 |

若参考机器无法达到目标，必须记录硬件、数据量、执行计划和瓶颈，再调整预算；禁止
通过减少验收数据量伪造达标结果。

## 12. 测试计划

### 12.1 单元测试

- 中文 CSV 表头和空值转换；
- 证券市场与内部键映射；
- 数值精度、成交量和金额单位；
- 分片边界、checkpoint 和重入；
- 复权因子推导、压缩和查找；
- Parquet 分区和 manifest 生成。

### 12.2 集成测试

- CSV → MySQL v2 → Repository → API；
- 同一批次重复执行不增加重复记录；
- 文件中断、MySQL 断开和进程重启后续传；
- 新旧 Repository 对同一证券返回一致结果；
- MySQL 版本与 Parquet 快照行数、范围和哈希一致；
- 增量更新后只重建受影响的快照分区。

### 12.3 性能与稳定性

- 10、100、1,000 和全市场四级导入基准；
- 冷缓存与热缓存查询基准；
- 导入期间并发执行单证券和列表查询；
- 两个全市场任务同时运行时的内存和磁盘压力；
- 大查询超时、取消和临时文件清理；
- 数据库连接池耗尽和磁盘空间不足。

只有满足以下条件后，才开始因子研究平台：

1. 全量历史数据已发布且通过基线核对；
